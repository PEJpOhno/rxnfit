# run_fit(p0=...) の辞書対応 仕様（docstring・コメント用）

## 概要

`ExpDataFitSci.run_fit(p0=...)` で、p0 に **辞書** を渡せるようにする。辞書のキーは **シンボリック速度定数名の文字列**（例: `"k1"`, `"k2"`）。リスト・タプルで渡す従来の指定もそのまま有効とする。

---

## 1. p0 の受け付け方

### 1.1 辞書で渡す場合

- **キー**: シンボリック速度定数名の **文字列**（例: `{"k1": 0.001, "k2": 0.002}`）。
- **値**: 数値（int または float）。その速度定数の初期値。
- **処理**: 内部で `get_symbolic_rate_const_keys()`（＝`param_info['symbolic_rate_consts']`）の **順序** に従い、リストに変換してから既存の最適化ロジックに渡す。
- **キーの検証**:
  - **余分なキー**: 辞書のキーが、定義されているシンボリック速度定数に含まれないものがある → **ValueError**。
    - メッセージ例: 「p0 に定義されていないシンボリックな速度定数が含まれています: {余分なキーの集合}. 使用可能: {symbolic_rate_const_keys}.」
  - **不足キー**: シンボリック速度定数のうち、辞書にキーが無いものがある → **ValueError**。
    - メッセージ例: 「p0 に次のシンボリックな速度定数が指定されていません: {不足キーのリスト}.」
  - 辞書の値が数値でない場合も **ValueError** とし、必要に応じてメッセージで「各値は数値である必要があります」と明示する。

### 1.2 リストまたはタプルで渡す場合

- **仕様**: 操作者が指定した **順番が正しい** とする（従来どおり）。
- **長さ**: `len(p0)` がシンボリック速度定数の数と一致しない場合は **ValueError**（現状どおり）。
  - メッセージ例: 「p0の長さ({len(p0)})がシンボリック速度定数の数({n_params})と一致しません。」

### 1.3 判定

- `isinstance(p0, dict)` が True のときは辞書として扱い、上記の検証とリスト変換を行う。
- それ以外はリストまたはタプルとして扱い、長さのみチェックする。

---

## 2. bounds（範囲の個別指定）との関係

- **bounds は従来どおりリストのまま** とする。変更しない。
- **順序**: `bounds[i]` は、i 番目のパラメータ（`symbolic_rate_const_keys` の i 番目）の (下限, 上限) を表す。
- p0 を **辞書** で渡した場合でも、bounds は **リスト** で、`symbolic_rate_const_keys` の順で渡す。
  - 例: シンボリックが `['k1','k2','k3']` のとき、  
    `p0={"k1": 0.001, "k2": 0.002, "k3": 0.001}` と  
    `bounds=[(1e-10, None), (1e-8, 100), (1e-6, None)]` の組み合わせでよい。
- **P0OptFit** は常に `bounds = [(self._bounds_per_param[i][0], None) for i in ...]` のように **リスト** で渡しており、順序は `self._symbolic_keys`（＝`get_symbolic_rate_const_keys()`）に一致しているため、**p0 を辞書対応しても不都合はない**。

---

## 3. P0OptFit との整合性

- **P0OptFit は p0 を常にリストで渡す**。
  - `_objective`: `p0 = [trial.suggest_float(name, ...) for name in self._symbolic_keys]` のあと `fit.run_fit(p0, ...)`。
  - `optimize`: `p0_best = [best.params[name] for name in self._symbolic_keys]` のあと `fit.run_fit(p0_best, ...)`。
- したがって、**run_fit が p0 の辞書を受け付けるようにしても、P0OptFit の呼び出しは変更不要**であり、**不都合は生じない**。
- 範囲の個別指定についても、P0OptFit は `param_bounds` で探索範囲を指定し、内部で `bounds` をリストに組み立てて run_fit に渡している。run_fit の `bounds` の型・順序は現状のままなので、**引数で範囲を個別に指定する場合も問題ない**。

---

## 4. docstring に書く内容（run_fit の p0 引数）

以下を run_fit の docstring に反映する。

- **p0** (list, tuple, or dict):  
  - **list / tuple**: シンボリック速度定数の初期値の並び。**操作者が指定した順番がそのままパラメータの順序**として扱われる。長さがシンボリック速度定数の数と一致しない場合は ValueError。
  - **dict**: シンボリック速度定数名（**文字列**）をキー、初期値を値とする辞書。例: `{"k1": 0.001, "k2": 0.002}`。キーは `get_symbolic_rate_const_keys()` で得られる名前と一致させること。定義されていないキーが含まれる場合、または必要なキーが欠けている場合は ValueError。値は数値であること。
- **bounds** (list, optional): 各パラメータの (下限, 上限)。**symbolic_rate_const_keys の順**で指定する。p0 を辞書で渡す場合も、bounds はこの順のリストで渡す。

---

## 5. コードコメントに書く内容

- run_fit の先頭で p0 をリストに正規化する処理を入れる場合、その直前に次のようなコメントを入れるとよい。
  - 「p0 が辞書の場合は、symbolic_rate_const_keys の順でリストに変換する。辞書のときはキー検証（余分・不足・値の型）を行い、問題があれば ValueError を出す。」
